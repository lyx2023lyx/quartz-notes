## 2.1 CPU应用程序执行模型

### 冯·诺依曼架构

```
┌─────────────────────────────────────────────────────────────┐
│                     冯·诺依曼架构                            │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│   ┌─────────┐      ┌─────────┐      ┌─────────┐            │
│   │  输入   │─────▶│  处理器  │◀────▶│   内存   │            │
│   └─────────┘      └────┬────┘      └─────────┘            │
│                         │                                   │
│                         ▼                                   │
│                    ┌─────────┐                              │
│                    │  输出   │                              │
│                    └─────────┘                              │
│                                                              │
│   处理器组成：                                               │
│   ├─ ALU（算术逻辑单元）：执行二进制运算                      │
│   ├─ 寄存器组：高速存储指令和数据                            │
│   ├─ 控制单元：控制指令读取                                  │
│   └─ 内存：存储所有指令和数据                                │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

**核心公式：消息传递时间**

$$
T_{message} = \alpha + \frac{N}{\beta}
$$

**白话解读**：
- **α（延迟）**：发送空消息的固定开销，包括软件开销、硬件开销、传输时间
- **N**：消息长度
- **β（带宽）**：通信介质容量

### 冯·诺依曼瓶颈

**问题**：处理器速度（~4GHz）远超内存传输速度，大量时间浪费在等待数据上。

**解决方案**：

| 方案 | 原理 | 效果 |
|------|------|------|
| **缓存** | 频繁使用的数据存入更快的存储区 | 减少主存访问 |
| **预取** | 提前将可能使用的数据放入缓存 | 隐藏延迟 |
| **多线程** | 等待时切换到其他线程 | 保持处理器繁忙 |

### 缓存系统详解

```
┌─────────────────────────────────────────────────────────────┐
│                      多级缓存结构                            │
│                     （以PS4为例）                            │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│   L1缓存（32KB） ◀── 3个时钟周期                             │
│      │                                                       │
│      ▼                                                       │
│   L2缓存（2MB） ◀── 30个时钟周期                             │
│      │                                                       │
│      ▼                                                       │
│   主存 ◀── 220+个时钟周期                                    │
│                                                              │
│   注：现代CPU的L1缓存分为L1 I$（指令）和L1 D$（数据）        │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

**缓存设计的核心原理：局部性**

| 类型 | 定义 | 示例 |
|------|------|------|
| **时间局部性** | 之前访问的数据很可能再次访问 | 循环中的循环变量 |
| **空间局部性** | 刚访问的数据附近数据可能马上被访问 | 数组顺序访问 |

**缓存行（Cache Line）**：
- 主存载入缓存的单位，通常64字节
- 使用LRU（最近最少使用）算法替换

### 预取技术

**目的**：减小缓存失效几率，提前将可能读取的数据载入缓存。

**预取模式**：

| 模式 | 适用场景 | 示例 |
|------|---------|------|
| **常数步幅** | 数据访问有固定步长 | `A[i*4]` 数组访问 |
| **随机访问** | ❌ 预取器不擅长 | `B[rand()%N]` 随机索引 |

**分支预测**：
- 记录历史数据，计算模式预测分支走向
- 排序后的数据分支易预测，随机数据难预测

## 2.2 并行计算架构

### 指令级并行（ILP）

**定义**：单处理器同时执行多条指令的能力。

**示例**：
```
1. e = a + b    ──┐
2. f = c + d    ──┼── 可并行执行（无依赖）
3. m = e * f    ◀── 依赖1和2的结果
```

**ILP值计算**：3条指令 / 2个周期 = **1.5**

#### 指令管线化（Pipelining）

经典5阶段划分：

| 阶段 | 缩写 | 功能 |
|------|------|------|
| 获取指令 | IF | 从L1I$获取指令（~1周期） |
| 解码+取操作数 | ID/RF | 解码指令，从寄存器取数 |
| 执行 | EX | ALU执行运算 |
| 内存访问 | MEM | 读取内存数据 |
| 写回 | WB | 结果写回寄存器 |

```
时钟周期：  1   2   3   4   5   6   7   8   9
指令1：     IF  ID  EX  MEM WB
指令2：         IF  ID  EX  MEM WB
指令3：             IF  ID  EX  MEM WB
指令4：                 IF  ID  EX  MEM WB
指令5：                     IF  ID  EX  MEM WB
            ↑
        从第5周期开始，每个周期完成1条指令
```

**处理指令依赖的3种方法**：

| 方法 | 原理 | 代价 |
|------|------|------|
| **管线气泡** | 发现依赖时插入空闲周期 | 延迟增加 |
| **操作数前移** | 直接传递前一指令结果 | 需检测依赖 |
| **乱序执行** | 不依赖的指令先执行 | 需重排序缓冲区 |

#### 操作数前移示例

```
原始（有气泡）：                    操作数前移优化：
ADD A B C   IF ID EX MEM WB       ADD A B C   IF ID EX MEM WB
SUB D C A       IF stall stall    SUB D C A       IF ID EX MEM WB
                    stall ID EX
                    ↑ 延迟3周期            ↑ 无延迟
```

### 线程级并行

**多线程原理**：
- 单个处理器内同时执行多个线程
- 线程共享处理器的计算单元、寄存器、缓存
- 一个线程等待时，切换到其他线程执行

**线程切换成本**：
- CPU：需保存/恢复寄存器状态
- GPU：每个线程拥有独立寄存器，切换成本≈0

#### 多线程技术对比

| 技术 | 英文 | 特点 |
|------|------|------|
| **块多线程** | Block Multithreading | 执行到遇到大延迟才切换 |
| **交叉多线程** | Interleaved Multithreading | 每个时钟周期切换不同线程 |
| **同时多线程** | SMT（超线程） | 同一周期执行多线程指令 |

**SMT vs 双核**：
- SMT：共享执行单元、L1缓存（芯片面积小）
- 双核：独立执行单元，仅共享L2以上缓存

### 处理器级并行

#### 多处理器架构类型

```
┌─────────────────────────────────────────────────────────────┐
│                    多处理器架构对比                          │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  对称多处理器（SMP）          非对称多处理器（ASMP）         │
│  ┌─────────┐ ┌─────────┐      ┌─────────┐                  │
│  │  核0    │ │  核1    │      │  主处理器 │                  │
│  │ L1缓存  │ │ L1缓存  │      │  (PPE)   │                  │
│  └────┬────┘ └────┬────┘      └────┬────┘                  │
│       └─────┬─────┘                │                        │
│        L2缓存/主存           ┌─────┼─────┐                  │
│                              ▼     ▼     ▼                  │
│                           ┌───┐ ┌───┐ ┌───┐                │
│                           │SPE│ │SPE│ │SPE│ （Cell架构）   │
│                           └───┘ └───┘ └───┘                │
│                                                              │
│  特点：所有核地位对等         特点：主从结构，主核协调从核    │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

#### 通信方式

| 方式 | 特点 | 问题 |
|------|------|------|
| **共享内存** | 多核共享主存 | 缓存一致性开销随核数增加 |
| **消息传递** | 通过网络连接独立计算机 | 延迟大，适合弱耦合计算 |

### 弗林分类法（Flynn's Taxonomy）

```
┌─────────────────────────────────────────────────────────────┐
│                      弗林分类法                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│                    单指令流        多指令流                 │
│                 ┌──────────┐    ┌──────────┐               │
│    单数据流     │   SISD   │    │   MISD   │               │
│                 │  串行    │    │  少见    │               │
│                 └──────────┘    └──────────┘               │
│                                                              │
│                 ┌──────────┐    ┌──────────┐               │
│    多数据流     │   SIMD   │    │   MIMD   │               │
│                 │  GPU核心 │    │  多核CPU │               │
│                 └──────────┘    └──────────┘               │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

| 类型 | 名称 | 说明 | 实例 |
|------|------|------|------|
| **SISD** | 单指令单数据 | 传统串行程序 | 单核CPU |
| **SIMD** | 单指令多数据 | 一条指令广播到多处理器，各处理不同数据 | GPU SP |
| **MISD** | 多指令单数据 | 理论完备性，无实际系统 | - |
| **MIMD** | 多指令多数据 | 各处理单元有自己的指令流和数据流 | 多核CPU |

**SIMD优势**：
- 只需一套指令解码逻辑
- 芯片面积小、成本低、能耗低
- 可工作在高时钟频率

## 2.3 GPU并行计算架构

### GPU与CPU的核心区别

```
┌─────────────────────────────────────────────────────────────┐
│                   CPU vs GPU 架构对比                        │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  CPU架构                      GPU架构                        │
│  ┌─────────────────┐         ┌─────────────────┐           │
│  │  复杂控制单元    │         │  简单控制单元    │           │
│  │  大容量缓存      │         │  小容量缓存      │           │
│  │  少量强大ALU    │         │  大量简单ALU    │           │
│  │  优化延迟       │         │  优化吞吐率     │           │
│  └─────────────────┘         └─────────────────┘           │
│                                                              │
│  目标：快速执行串行代码       目标：大规模并行计算           │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### GPU内存结构

```
┌─────────────────────────────────────────────────────────────┐
│                     GPU内存层级                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│   寄存器（Register）◀── 1周期，~8TB/s                        │
│      │                                                       │
│      ▼                                                       │
│   共享内存（Shared Memory）◀── 1-32周期，~1.5TB/s           │
│      │  （L1缓存，SM内部）                                    │
│      ▼                                                       │
│   常量/纹理内存（Constant/Texture）◀── 400-600周期          │
│      │  （带缓存，适合随机访问）                              │
│      ▼                                                       │
│   全局内存（Global Memory）◀── 400-600周期，~200MB/s        │
│      │  （GPU主存，通过PCI-E与CPU连接）                       │
│      ▼                                                       │
│   CPU内存（通过PCI-E，~8GB/s）                               │
│                                                              │
│   关键区别：GPU使用程序托管内存模型（非硬件自动缓存）        │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

#### 全局内存

- GPU的主内存，CPU和GPU都可读写
- 通过PCI-E总线与CPU连接（PCI-E 3.0: ~8GB/s）
- 使用流传输可实现数据传输与内核执行重叠

#### 共享内存

**特点**：
- 位于SM内部，延迟极低
- 使用Bank-Switched架构分成多个存储体（banks）
- 理想情况：32个线程访问32个不同bank，可同时执行

**存储体冲突**：
- 多个线程访问同一bank → 需要排队
- 解决方案：确保线程访问不同bank或同一地址（广播）

```
共享内存Bank结构：
┌────┬────┬────┬────┬────┐
│B0  │B1  │B2  │... │B31 │  ← 32个存储体
└────┴────┴────┴────┴────┘
  ↑    ↑    ↑         ↑
 T0   T1   T2   ...  T31  ← 线程束中的32个线程
 
理想：每个线程访问不同Bank → 同时执行
冲突：多个线程访问同一Bank → 排队等待
```

#### 寄存器

**GPU寄存器特点**：
- 数量巨大（如GK110：65,536个32位寄存器）
- 数据不会自动写回全局内存，直到线程结束
- 每个线程拥有独立寄存器，切换无成本

### 流处理器族（SM）架构

以NVIDIA Kepler架构（GK110/210）为例：

```
┌─────────────────────────────────────────────────────────────┐
│                   Kepler SMX架构                             │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│   一个GPU = SMX阵列（GK110有15个SMX）                        │
│                                                              │
│   每个SMX包含：                                              │
│   ├─ 192个流处理器（SP）                                     │
│   ├─ 65,536个32位寄存器                                      │
│   ├─ 共享内存（L1）                                          │
│   ├─ 特殊功能单元（SFU）                                     │
│   └─ 加载/存储单元                                           │
│                                                              │
│   每个SP每个时钟周期执行1条指令                              │
│   理想峰值：192线程 × 15SMX = 2,880线程/周期                 │
│                                                              │
│   计算性能：高达1 TFlop/s（每秒万亿次浮点运算）              │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

**SIMD执行模型**：
- 一个SMX内所有SP执行相同指令
- 单条指令广播到所有SP
- 每个SP处理不同的数据

### 延迟隐藏（Latency Hiding）

**GPU的延迟隐藏策略**：

| 技术 | 实现方式 | 效果 |
|------|---------|------|
| **大量线程** | GK110每SMX支持2048线程（192SP） | 总有线程可执行 |
| **大寄存器文件** | 65,536+寄存器 | 线程切换成本≈0 |
| **内存合并** | 连续线程访问连续地址 | 减少内存事务 |

```
延迟隐藏示意图：

传统CPU（少量线程）：               GPU（大量线程）：
┌────┬────┬────┬────┐            ┌────┬────┬────┬────┐
│ T0 │等待│ T0 │等待│            │ T0 │ T1 │ T2 │ T3 │
└────┴────┴────┴────┘            │ T4 │ T5 │ T6 │ T7 │
  ↑  ↑                            │ .. │ .. │ .. │ .. │
  计算 内存读取                      │Tn  │T0  │T1  │T2  │
                                    └────┴────┴────┴────┘
                                     ↑  ↑  ↑  ↑
                                     总有线程在执行
```

### 内存合并（Memory Coalescing）

**原理**：当连续线程访问连续对齐的内存地址时，多个请求合并为一次内存事务。

```
内存合并示例：

线程束（32线程）访问全局内存：

可合并（理想）：                    不可合并（离散）：
┌────┬────┬────┬────┐            ┌────┐    ┌────┐    
│ T0 │ T1 │ T2 │... │            │ T0 │    │ T2 │    
│addr│addr│addr│    │            │addr│    │addr│    
│ 0  │ 4  │ 8  │... │            │100 │    │108 │    
└────┴────┴────┴────┘            └────┘    └────┘    
  │    │    │                       │         │       
  └────┼────┘                       └────┬────┘       
       ▼                                  ▼           
  1次内存事务                        多次内存事务      
  （32字节~128字节）                 （每个线程单独）  
```

**对齐要求**：
- 数据类型需与内存系统支持的大小对齐
- 内置类型（int, float, float2, float4）自动对齐
- 自定义类型可使用`__align__`标识符强制对齐

### 分支处理

**GPU分支特点**：
- 无复杂分支预测单元
- SIMD架构每次只获取一条指令
- 一个线程束执行完一个分支后，再执行另一个分支

```
分支执行示例：

条件：if (x > 0) A else B

理想情况（分支连续）：              糟糕情况（分支交错）：
┌─────────────────┐                ┌─────────────────┐
│ T0-T15: x>0    │ 执行A           │ T0: x>0        │ 执行A
│ T16-T31: x≤0   │ 执行B           │ T1: x≤0        │ 执行B
│                  │                │ T2: x>0        │ 执行A
│ 两个分支并行执行  │                │ T3: x≤0        │ 执行B
│ 利用率100%       │                │ ...            │ ...
└─────────────────┘                │ 频繁切换分支   │
                                    │ 大量线程闲置   │
                                    └─────────────────┘
```

**优化建议**：
- 尽量保证分支连续性
- 按条件对数据进行排序
- 避免线程束内分支分布不均匀
