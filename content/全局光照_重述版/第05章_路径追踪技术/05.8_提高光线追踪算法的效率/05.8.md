# 5.8 提高光线追踪算法的效率

## 核心总结
光线追踪最大的开销是找光线和物体的交点，通过各种加速结构、并行计算、缓存优化等手段，可以大幅提升渲染速度。

---

## 光线追踪的瓶颈在哪？

### 性能分析

**时间占比**：
- 光线-物体相交：60-80%
- 着色计算：10-20%
- 内存访问：10-20%

**关键问题**：
一条光线要和场景中所有三角形检查相交吗？
- 100万三角形 × 100万像素 × 多次反弹 = 天文数字！

**必须加速**！

---

## 加速结构

### 1. 包围盒层次结构（BVH - Bounding Volume Hierarchy）

**核心思想**：
- 用简单盒子包复杂物体
- 先和盒子检查，击中了才检查里面的物体

**构建过程**：
```
1. 所有物体包在一个大盒子里（根节点）
2. 沿某个轴（x/y/z）分成两半
3. 分别构建左右子树
4. 直到盒子内物体少于阈值
```

**遍历过程**：
```
函数 Traverse(光线, 节点):
    如果 光线和节点盒子不相交: 返回 无交点
    
    如果 节点是叶子:
        检查节点内所有物体
        返回 最近交点
    
    // 是内部节点
    左结果 = Traverse(光线, 左子节点)
    右结果 = Traverse(光线, 右子节点)
    返回 最近的(左结果, 右结果)
```

**优点**：
- 构建相对简单
- 遍历快（cache友好）
- 现代GPU硬件支持

**对应图片讲解**：
- 图片展示了BVH结构
- 大盒子包小盒子
- 叶子节点包含实际物体

---

### 2. KD树（K-Dimensional Tree）

**核心思想**：
- 空间分割（不是物体分割）
- 沿着x/y/z轴轮流分割空间

**构建**：
- 选择分割轴（如x→y→z→x...）
- 选择分割位置（中位点）
- 递归分割

**遍历**：
- 光线穿过空间单元
- 只检查单元内的物体
- 按光线方向顺序遍历

**vs BVH**：
| 特性 | BVH | KD树 |
|------|-----|------|
| 分割对象 | 物体 | 空间 |
| 构建 | 快 | 慢 |
| 遍历 | 快 | 中等 |
| 更新 | 容易 | 难 |
| 适用 | 动态场景 | 静态场景 |

---

### 3. 均匀网格（Uniform Grid）

**核心思想**：
- 把空间切成均匀小格子
- 每个格子记录里面的物体

**光线遍历**：
- 3D DDA算法（类似Bresenham）
- 沿光线方向逐个格子检查

**优点**：
- 实现简单
- 遍历非常快

**缺点**：
- 物体分布不均匀时效率低
- 空旷区域浪费检查

---

### 4. 八叉树（Octree）

**核心思想**：
- 自适应细分空间
- 物体密的地方多分，疏的地方少分

**结构**：
- 每个节点8个子节点（2×2×2）
- 递归细分直到满足条件

**适用**：
- 稀疏场景
- 需要自适应细分的场景

## 光线排序（Ray Sorting）

### 问题

**内存访问不连续**：
- 光线随机发射
- 访问的物体分散在内存各处
- cache miss严重

### 解决：光线打包

**原理**：
- 相似方向的光线放在一起处理
- 访问相似的物体
- 提高cache命中率

**方法**：
1. **空间排序**：把靠近的光线分组
2. **方向排序**：把方向相似的光线分组
3. **材质排序**：把命中相同材质的光线分组

**实现**：
```cpp
// 先收集所有光线
vector<Ray> rays;
for each pixel:
    rays.push_back(GenerateRay(pixel));

// 按方向排序
sort(rays, CompareByDirection);

// 批量处理
for each batch of rays:
    ProcessBatch(batch);
```

## 并行化

### CPU并行

**多线程**：
- 每个线程处理一部分像素
- 共享加速结构
- 注意同步问题

**SIMD**：
- SSE/AVX指令
- 一条指令处理4/8条光线
- 大幅提升效率

### GPU并行

**CUDA/OpenCL**：
- 成千上万个线程
- 每个线程处理一条光线
- 适合数据并行的任务

**RT Core（NVIDIA）**：
- 专门硬件加速BVH遍历
- 专门硬件加速光线-三角形相交
- 比纯软件快10倍以上

**对应图片讲解**：
- 图片展示了GPU并行架构
- 大量线程同时工作
- 共享内存加速访问

## 早期剔除

### 1. 背面剔除

**原理**：
- 三角形的背面不会朝向相机
- 只检查正面

**实现**：
```cpp
if (dot(ray.direction, triangle.normal) > 0)
    return; // 背面，跳过
```

### 2. 可见性剔除

**视锥体剔除**：
- 不在相机视锥内的物体不需要渲染

**遮挡剔除**：
- 被其他物体完全挡住的物体不需要渲染

### 3. 距离剔除

**原理**：
- 太远的小物体看不见
- 根据距离剔除

## 延迟渲染（Deferred Shading）

### 核心思想

**先找交点，再着色**：
1. 第一阶段：只找最近交点，记录位置、法线、材质ID
2. 第二阶段：用G-Buffer信息着色

**好处**：
- 避免重复光线相交计算
- 光源计算可以在屏幕空间做
- 适合多光源场景

### G-Buffer

**存储内容**：
- 位置（World Position）
- 法线（Normal）
- 反照率（Albedo）
- 材质参数（Roughness, Metallic）
- 深度（Depth）

---

## 内存优化

### 1. 紧凑存储

**三角形数据**：
- 位置：3×float（12字节）
- 可以压缩为half float或整数

**BVH节点**：
- 包围盒：6×float
- 子节点索引：2×int
- 紧凑排布，减少内存占用

### 2. 缓存优化

**节点访问模式**：
- 子节点放在父节点附近
- 提高cache命中率

**宽BVH（Wide BVH）**：
- 每个节点4或8个子节点
- 减少遍历深度
- 更适合SIMD

---

## 智能采样

### 1. 自适应采样

**原理**：
- 噪点多的地方多采样
- 平滑的地方少采样

**实现**：
- 先渲染低分辨率预览
- 计算局部方差
- 方差大的地方增加采样

### 2. 重要性采样优化

**BRDF采样**：
- 根据材质特性采样
- 镜面材质朝反射方向
- 漫反射均匀采样

**光源采样**：
- 大光源多采样
- 亮光源多采样
- 近光源多采样
